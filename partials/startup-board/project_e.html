2. 비동기 Async

 사실 메일 전송같은 기능들은 그냥 동기화를 개발하면 문제가 많다. 왜냐하면 메일전송은 긴시간 이 소요된다. 그렇기 때문에 많은 Request 요청시 그 요청에 대기시간이 길어지고, 스레드를 할당 받지 못하는 경우가 많이 생길 수 있다. 그렇기 때문에 멀티스레드를 이용한 비동기로 구현을 해야한다. 사실 예전에는 스프링에서 제공하는 TaskExcutor를 이용해서 스레드풀을 만들어서 메일 기능 마다 사용했지만 요번에는 조금 다르게 그Api 전체를 비동기로 했다.

먼저 스프링에서 제공하는 TaskExcutor를 기본으로 스레드 풀을 생성한다. 


@Override
public void configureAsyncSupport(AsyncSupportConfigurer configurer) {
	// TODO Auto-generated method stub

    ThreadPoolTaskExecutor taskExecutor = new ThreadPoolTaskExecutor();
    taskExecutor.setCorePoolSize(10);
    taskExecutor.setMaxPoolSize(20);
    taskExecutor.setQueueCapacity(50);
    configurer.setTaskExecutor(taskExecutor);		
}

스프링 3.2 에서 제공하는 Callable을 리턴 타입으로 지정 Callable은 인터페이스를 기본으로 하기 때문에 람다식으로 구현하였다.


@Autowired
AccountService accountService;

@RequestMapping(value ="/", method = RequestMethod.Post)
public @ResponseBody Callable Account set(@ModelAttribute Account account){
			
	return () -> {	
		return accountService.findByEmail(account.getEmail());
	};
}

3. ORM

 orm으로 JPA를 사용했다. JPA는 모델링을 잘 이해하고 사용하는 것이 중요하다는 문서가 있다. 잘 못 관계를 맺고 사용하거나 모델링을 잘못했을때 이슈가 많다고 들었다. 하지만 이서버는 많은 DB모델링이 필요하지않기 때문에 연습 삼아 사용하였다. 내가 사용하면서 느낀 JPA의 장점은 다음과 같았다.


1. 쿼리 최적화

JPA는 실행되는 쿼리를 알아서 최적화 해준다고한다. 그래서 테스트겸 똑같은 쿼리를 두번 작성해 보았더니 실제로 DB에 쿼리를 한번만 실행시켰다. 프로그램을 읽고 쿼리를 실행시키는 과정에서 최적화를 하는 것 이다. 그래도 혹시 몰라 매번 쿼리의 로그를 남겼다.


2. 매번 쿼리를 작성해야하는 번거로움에서 해방

사실 JPA에 가장 큰매력을 느꼈던 점이다. Domain모델을 잘 설정해 두면 간단한 crud는 set, find, delete로 처리 할 수 있다. Domain간에 관계설정을 통한 조인도 쉽다.(하지만 잘 못 사용하면 독이 될 수 있다.) 그리고 필요에 따라서 쿼리를 직접 작성할 수 도 있다.


JPA가 매우 편한것은 사실이지만 쿼리를 몰라서는 안된다. Domain에서 Join설정을 해놓고 Join이 필요없는 데이터를 뽑아온다던지, 잘못된 설정으로 성능이슈가 발생할 가능성이 크다고 생각한다. JPA를 사용하려면 기존보다 더많은 DB공부가 필요한 것 같다. 그리고 영속성 비영속성 상태에 대해서도 알아둘 필요가 있다. 틈틈히 "DDD"에 대한 공부를 해봐야겠다.


마지막으로 api는 항상 설명서를 문서화 해놓는것이 중요하다!!
